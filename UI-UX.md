# GridMove UI / UX 说明

本文档描述 GridMove 的 UI 控件交互逻辑、信息分组方式、以及未来可视化设置界面的组织原则。

本文档不负责记录程序级细节、底层事件流、或配置文件兼容策略。这些内容请参考 `APP-DESIGN.md`。

## 1. 目标

GridMove 的 UI 目标是：

- 让用户快速理解当前运行状态
- 让常用操作可以直接执行
- 让设置项按关系组织，而不是按底层字段平铺
- 尽量使用原生控件和标准 macOS 交互

## 2. 当前界面

当前已经存在的界面有：

- 菜单栏菜单
- 系统辅助功能权限请求

未来会补充：

- 设置窗口

## 3. 文案规则

可见文案遵循以下规则：

- 默认使用 sentence case
- 尽量短，优先描述用户可理解的动作或状态
- 同一动作在不同界面尽量使用同一文案

当前可见文案的统一来源是：`Sources/GridMove/Support/UICopy.swift`。

## 4. 菜单栏菜单

### 4.1 正常状态

辅助功能权限可用时，菜单栏菜单包含四组内容：

1. 运行总开关
2. 触发方式与触发偏好
3. 当前布局组相关操作
4. 配置与应用级操作

当前菜单项顺序为：

- `Enable`
- 分隔线
- `Middle mouse drag` 或 `Mouse button <n> drag`
- `Modifier + left mouse drag`
- `Prefer layout mode`
- 分隔线
- `Layout group` > 二级菜单选择 group
- 分隔线
- 当前布局组下的布局操作
- 分隔线
- `Reload`
- `Customize... ↗`
- `Launch at login`
- 分隔线
- `Quit`

这些项的关系应当保持明确：

- `Enable` 是应用级总开关，控制 GridMove 是否处于启用状态

- `Middle mouse drag`、`Modifier + left mouse drag`、`Prefer layout mode` 属于同一组，都是“拖动触发行为”的设置

- `Reload`、`Customize... ↗`、`Launch at login`、`Quit` 属于配置与应用行为，不应与布局操作混在一起

### 4.2 菜单项行为

- `Enable`
  - 控制整个应用是否启用
  - 是即时开关
  - 状态变化应立即反映到菜单、运行时监听、和持久化配置

- `Middle mouse drag` / `Mouse button <n> drag`
  - 控制鼠标按钮拖动触发是否启用
  - 菜单文案需要反映当前按钮编号
  - 这是触发方式组的一部分，不是独立功能块

- `Modifier + left mouse drag`
  - 控制修饰键加左键拖动触发是否启用
  - 它与修饰键组合配置属于同一条交互链

- `Prefer layout mode`
  - 控制拖动进入后默认是布局模式还是自由移动模式
  - 它描述的是触发后的默认模式，因此应与触发方式放在同组

- `Layout group`
  - 用来切换当前激活的布局组
  - 这是后续布局动作的上下文选择器
  - 不应当被当作普通动作项

- 布局操作
  - 文案由当前布局组中的可见布局生成
  - 同一组中布局的顺序应与实际应用顺序一致
  - 如存在快捷键，在菜单中显示对应快捷键信息
  - 点击后直接对当前目标窗口执行动作

- `Reload`
  - 重新从磁盘加载配置
  - 它属于维护操作，不属于布局或触发控制

- `Customize... ↗`
  - 打开配置目录
  - 它是配置入口，不是运行时状态切换

- `Launch at login`
  - 控制开机自启
  - 它对应 `general.launchAtLogin`
  - 这是系统集成能力，不属于拖动触发组

- `Quit`
  - 退出应用

### 4.3 权限缺失状态

辅助功能权限缺失时，菜单栏菜单退化为单项：

- `Get accessibility access`

此时其它菜单项不展示。原因是这些功能都依赖辅助功能权限，继续展示会让状态和能力边界不清楚。

`Get accessibility access` 的行为：

- 点击后重新触发系统辅助功能权限请求
- 在真正获得权限前，菜单保持单项状态
- 权限恢复后，正常菜单自动恢复

## 5. 辅助功能权限请求

GridMove 当前不实现自定义的权限引导窗口，直接使用系统权限请求。

交互规则：

- 应用启动时，如缺少辅助功能权限，直接触发一次系统请求
- 如果用户拒绝，应用继续驻留在菜单栏，但菜单只保留 `Get accessibility access`
- 如果运行中权限从已授权变为未授权，也应切回同样的受限状态
- 用户在受限状态下点击 `Get accessibility access`，应再次触发系统请求

权限未授予时的 UI 结果：

- 拖动触发不可用
- 快捷键监听不可用
- 布局相关菜单项不可用且不展示
- 配置与应用级菜单项也不展示

## 6. 设置窗口

> 当前尚未完整实现。此处描述的是未来设置界面的交互组织原则。

### 6.1 总体结构

设置界面应当由两个部分构成：

- 全局切换 tab
- 当前 tab 对应的设置详情

通用规则：

- tab 只负责切换当前设置领域
- 设置详情负责展示该领域下有关联的控件组
- 同一组控件应围绕一个明确目标组织，而不是按底层字段逐个罗列
- 除特别说明外，设置修改后应即时生效

当前 tab 规划为：

- `General`
- `Layouts`
- `Appearance`
- `Hotkeys`
- `About`

## 7. General

`General` 负责应用级运行方式和触发方式配置。

它至少应包含以下几个设置组。

### 7.1 运行方式组

这一组描述应用整体如何运行：

- `Enable`
- `Launch at login`

这两个控件的关系是：

- `Enable` 控制当前这次运行是否生效
- `Launch at login` 控制系统层面的自动启动
- 两者都属于应用级状态，但作用层级不同，因此应归在一组而不是拆散

### 7.2 拖动行为组

这一组描述“如何进入拖动交互，以及进入后默认处于什么模式”：

- `Middle mouse drag`
- `Mouse button number`
- `Modifier + left mouse drag`
- `Modifier groups`
- `Prefer layout mode`

这些控件的关系是：

- `Middle mouse drag` 与 `Mouse button number` 是一条配置链，前者控制是否启用，后者定义具体按钮
- `Modifier + left mouse drag` 与 `Modifier groups` 是另一条配置链，前者控制是否启用，后者定义允许的组合
- `Prefer layout mode` 描述的是上述触发方式进入后的默认行为，因此也属于同组

`Modifier groups` 应支持：

- 展示当前组合列表
- 添加组合
- 删除组合
- 每个组合由一个或多个修饰键构成

### 7.3 排除项组

这一组描述哪些窗口不应由 GridMove 处理：

- `Excluded bundle IDs`
- `Excluded window titles`

这两个列表属于同一个目标：定义排除规则。它们应以统一的交互方式呈现，例如都支持：

- 查看已有条目
- 添加条目
- 删除条目

## 8. Layouts

`Layouts` 负责维护布局组、布局集和布局本身。

它不应只展示一个平铺的布局列表，而应反映三层关系：

1. 布局组
2. 布局组内的显示器目标集
3. 目标集内的布局列表与布局详情

### 8.1 布局组层

布局组层至少需要表达：

- 当前有哪些布局组
- 当前激活的是哪一个布局组
- 该布局组是否参与组循环
- 该布局组适用的显示器

因此，布局组相关控件应围绕以下关系组织：

- 组名称
- `includeInGroupCycle`
- 组选中状态或当前编辑目标

### 8.2 显示器目标集层

每个布局组下面可能有多个目标集。一个目标集至少包含：

- 目标显示器
- 该显示器目标下的布局列表

显示器目标控件应表达以下几种语义：

- `all`
- `main`
- 一个具体显示器
- 多个具体显示器

当使用具体显示器时，UI 应向用户展示易懂的显示器名称，但内部仍使用持久 ID。

### 8.3 布局列表与排序

布局列表应支持：

- 查看当前目标集下的布局
- 调整顺序
- 新增布局
- 删除布局

顺序不是纯视觉问题，它直接影响：

- 菜单顺序
- layout index
- 循环布局的顺序

因此排序控件应与布局列表放在一起，而不是放入单个布局详情里。

### 8.4 布局详情

单个布局详情至少包含：

- `Name`
- `Include in menu`
- `Include in layout index`
- 网格尺寸
- `Window layout`
- `Trigger regions`

这些控件应按关系组织：

- `Name` 负责用户识别
- `Include in menu` 和 `Include in layout index` 共同定义这个布局在哪些入口可见
- 网格尺寸决定下面两类可视化编辑的坐标系统
- `Window layout` 和 `Trigger regions` 是同一布局的两个视图，不应被拆成互不相关的页面

## 9. Appearance

`Appearance` 负责预览和覆盖层样式。

它应包含两个主要设置组：

- `Window highlight`
- `Trigger overlay`

这两个组的关系是：

- 都属于叠加层样式
- 但目标不同，一个用于窗口高亮，一个用于触发区域

`Window highlight` 组至少包括：

- 是否显示
- 填充透明度
- 描边宽度
- 描边颜色
- layout gap

`Trigger overlay` 组至少包括：

- 是否显示
- 填充透明度
- 间距
- 描边颜色

Appearance 页面中的预览应展示用户当前设置的结果，但预览本身不是运行时状态切换入口。

## 10. Hotkeys

`Hotkeys` 负责动作和快捷键绑定关系。

它应表达两个维度：

- 有哪些动作
- 每个动作绑定了哪些快捷键

动作不应只包含固定的全局动作，还应包含来自当前布局配置的布局动作。

因此这个页面至少应覆盖：

- `Apply next layout`
- `Apply previous layout`
- 当前布局列表派生出的 `Apply <layout name>`

一个动作可以绑定多个快捷键。交互上应支持：

- 查看现有绑定
- 添加绑定
- 删除绑定
- 临时为空

动作文案应与菜单栏使用相同命名体系，避免同一动作在不同界面有不同说法。

## 11. About

`About` 负责展示稳定的产品信息。

当前至少应包含：

- 版本号

未来如扩展更新检查、更新日志等，也应继续放在这一页，而不是分散到其它 tab。

## 12. 共享预览组件

布局相关页面应复用一套共享预览组件。

这套组件的职责是：

- 展示窗口布局
- 展示触发区域
- 展示 Grid 切分细节并以 Grid 为单位调节窗口布局和触发区域

这套组件：
- 必须展示菜单栏和屏幕区域
- 应当忠实反映 Appearance 配置的样式

同一套组件在不同页面中的区别，应由上层页面决定，而不是由组件自己改变数据模型。因此上层页面可以控制：

- 只看 `Window layout`
- 只看 `Trigger regions`
- 同时看两者
- 是否允许编辑

共享预览组件需要保持以下一致性：

- 同一数据在不同页面的几何结果一致
- 屏幕区域和菜单栏区域使用同一套基本坐标逻辑
- 可编辑与只读是模式差异，不应导致结构差异
- 预览组件的比例应当和主显示器保持一致，无法缩放。
